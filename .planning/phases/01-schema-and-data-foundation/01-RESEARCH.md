## Summary

Phase 1 is a pure data-layer phase: two new Prisma models, one migration, and two utility files. There is no UI work and no changes to existing models. The existing schema (`Student`, `Session`, `Invoice`, `TutorProfile`) is additive-only ã new tables are appended, nothing is altered or dropped.

The central design decision already locked in STATE.md is just-in-time charge generation: charges are not created eagerly when a subscription is saved, but on demand when `ensureChargeForMonth` is called. This means `PlatformCharge` rows are created at invoice-generation time (Phase 3), not at subscription-creation time (Phase 2). The unique constraint `@@unique([subscriptionId, month, year])` on `PlatformCharge` is the idempotency mechanism ã Prisma's `upsert` or `createMany({ skipDuplicates: true })` exploits it so calling the helper twice is a no-op.

The database is Supabase PostgreSQL. The schema already has `directUrl = env("DIRECT_URL")` configured, which is the correct setup for migrations against Supabase (the pooled `DATABASE_URL` cannot run DDL). This means `prisma migrate deploy` will work correctly when `DIRECT_URL` points to the direct (non-pooled) connection string.

**Primary recommendation:** Add `PlatformSubscription` and `PlatformCharge` as new Prisma models, generate and commit the migration with `prisma migrate dev`, then write `lib/platform-charges.ts` with `addMonths` and `ensureChargeForMonth`. Do not modify any existing model.

### Pattern 1: Prisma Model Design ã PlatformSubscription

**What:** Represents one active (or deactivated) subscription for a student. Stores the monetary amount, billing method, duration, and when it started.

**When to use:** One row per subscription contract. A student can have at most one active subscription at a time (enforced by application logic in Phase 2, not a DB constraint in Phase 1).

```prisma
// Billing method: "invoice" means fee appears on the monthly PDF invoice.
// "direct" means student pays by bank transfer; tutor marks it manually (Phase 4).
model PlatformSubscription {
  id            String          @id @default(cuid())
  studentId     String
  student       Student         @relation(fields: [studentId], references: [id])
  amountCHF     Float
  billingMethod String          // "invoice" | "direct"
  durationMonths Int            // 1 or 6
  startMonth    Int             // 1-12
  startYear     Int
  active        Boolean         @default(true)
  charges       PlatformCharge[]
  createdAt     DateTime        @default(now())
  updatedAt     DateTime        @updatedAt
}
```

**Confidence:** HIGH ã mirrors the existing `Session` and `Invoice` model conventions (cuid IDs, `studentId` FK, `month`/`year` as Int, `createdAt`).

### Pattern 2: Prisma Model Design ã PlatformCharge

**What:** One row per (subscription, month, year) combination. Created just-in-time. The unique constraint makes creation idempotent.

```prisma
model PlatformCharge {
  id             String               @id @default(cuid())
  subscriptionId String
  subscription   PlatformSubscription @relation(fields: [subscriptionId], references: [id])
  month          Int
  year           Int
  amountCHF      Float
  paidAt         DateTime?            // null = unpaid; set by Phase 4
  createdAt      DateTime             @default(now())

  @@unique([subscriptionId, month, year])
}
```

**Confidence:** HIGH ã `@@unique` constraint mirrors `Invoice`'s `@@unique([studentId, month, year])` exactly.

### Pattern 4: addMonths Utility

**What:** Pure function that takes a `(startMonth, startYear, offsetMonths)` and returns `{ month, year }` handling year-wrap correctly.

**When to use:** Generating the sequence of charge months for a 6-month subscription.

```typescript
// lib/platform-charges.ts
export function addMonths(
  startMonth: number,
  startYear: number,
  offset: number
): { month: number; year: number } {
  // month is 1-indexed (1 = January, 12 = December)
  const totalMonths = startMonth - 1 + offset; // 0-indexed total
  const year = startYear + Math.floor(totalMonths / 12);
  const month = (totalMonths % 12) + 1;
  return { month, year };
}
```

**Year-wrap example:** `addMonths(10, 2025, 0)` ã `{month:10, year:2025}`, `addMonths(10, 2025, 3)` ã `{month:1, year:2026}`. Verified manually: October (month 10) + 3 months = January (month 1) of next year. The formula `totalMonths = (10-1) + 3 = 12`; `Math.floor(12/12) = 1` year added; `12 % 12 = 0` ã month = 1. Correct.

**Confidence:** HIGH ã pure arithmetic, verifiable by hand.

### Pattern 5: ensureChargeForMonth

**What:** Idempotent function that creates a `PlatformCharge` row if it doesn't already exist for the given subscription/month/year. Uses Prisma's `upsert` with the unique constraint as the where-clause.

```typescript
// lib/platform-charges.ts
import { prisma } from "@/lib/prisma";

export async function ensureChargeForMonth(
  subscriptionId: string,
  month: number,
  year: number,
  amountCHF: number
): Promise<void> {
  await prisma.platformCharge.upsert({
    where: { subscriptionId_month_year: { subscriptionId, month, year } },
    update: {}, // no-op if already exists
    create: { subscriptionId, month, year, amountCHF },
  });
}
```

**Idempotency:** Calling this twice with the same arguments: first call creates the row; second call hits the `where` match and executes the empty `update: {}` ã the row count stays at 1.

**Confidence:** HIGH ã standard Prisma upsert pattern; `update: {}` for idempotent creation is a documented pattern.

### Pattern 6: getInvoicePayload Integration Point

**What:** Phase 3 will call `ensureChargeForMonth` inside `getInvoicePayload`. Phase 1 only needs to design the helper's signature so Phase 3 can call it without breaking existing behavior for students without subscriptions.

**Key constraint:** `getInvoicePayload` must remain unchanged after Phase 1. The helper must be callable optionally ã if a student has no active invoice-billed subscription for the given month, it is simply not called and the payload is identical to today's output.

**Design implication:** `ensureChargeForMonth` should be a standalone exported function, not side-effectfully auto-invoked on import. Phase 3 will wire the call.

**Confidence:** HIGH ã directly stated in Phase 1 success criteria: "can be called from `getInvoicePayload` without breaking existing invoice generation for students without subscriptions."

### Anti-Patterns to Avoid

- **Modifying existing models:** Do not alter `Student`, `Session`, `Invoice`, or `TutorProfile` columns. Adding the `subscriptions` relation field to `Student` is virtual and safe ã it generates no ALTER COLUMN.
- **Eager charge generation:** Do not create all 6 charges when a subscription is saved. Just-in-time is the locked decision (STATE.md).
- **Using `db push` instead of `migrate dev`:** `prisma db push` does not create a migration file; `prisma migrate dev` generates `migration.sql` which is required for `prisma migrate deploy` on Supabase.
- **Running migrations on pooled connection:** Supabase's `DATABASE_URL` is a pooled PgBouncer URL and cannot run DDL. `DIRECT_URL` must be set and `prisma migrate deploy` uses it via the `directUrl` in schema.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Idempotent row creation | Custom SELECT then INSERT logic | Prisma `upsert` with `@@unique` | Race condition between SELECT and INSERT; Prisma upsert is a single atomic DB operation |
| Migration tracking | Manual SQL scripts with version table | `prisma migrate dev` / `prisma migrate deploy` | Prisma tracks applied migrations in `_prisma_migrations`; manual scripts skip this and cause "drift" errors |
| Month arithmetic | Date library (date-fns, dayjs) | Pure integer arithmetic in `addMonths` | No new dependency needed; arithmetic on 1-indexed month integers is simple and fully testable without date parsing |

**Key insight:** The combination of `@@unique` + Prisma `upsert` is the idiomatic solution to "create if not exists" in Prisma. It is atomic and avoids any race condition that a manual check-then-create pattern would have.

## Common Pitfalls

### Pitfall 1: Supabase Migration Connection

**What goes wrong:** Running `prisma migrate deploy` with the pooled `DATABASE_URL` (PgBouncer) fails with a connection error or migration is silently skipped.

**Why it happens:** PgBouncer operates in transaction pooling mode which does not support the `SET` commands Prisma migration engine uses for session-level DDL.

**How to avoid:** Verify `.env.local` contains both `DATABASE_URL` (pooled, for runtime queries) and `DIRECT_URL` (direct Supabase connection, for migrations). The schema already has `directUrl = env("DIRECT_URL")` ã confirm the env var is populated before running the migration.

**Warning signs:** Error message mentioning "prepared statements" or "connection refused on port 6543" (PgBouncer port) during migrate.

### Pitfall 2: Prisma Client Not Regenerated

**What goes wrong:** TypeScript types for `PlatformSubscription` and `PlatformCharge` don't exist after schema edit; `prisma.platformSubscription` is undefined at runtime.

**Why it happens:** The Prisma client is generated from `schema.prisma` ã editing the schema does not automatically regenerate it.

**How to avoid:** After running `prisma migrate dev`, the client is regenerated automatically. If using `prisma migrate deploy` (production), run `prisma generate` explicitly before starting the server. The project's `postinstall` script already runs `prisma generate` on `npm install`.

**Warning signs:** TypeScript error `Property 'platformSubscription' does not exist on type 'PrismaClient'`.

### Pitfall 3: Month Off-By-One in addMonths

**What goes wrong:** A subscription starting in month 10 with duration 6 generates charge months that are incorrect (e.g., months 11-16 instead of 10-12, 1-3).

**Why it happens:** Mixing 0-indexed and 1-indexed month arithmetic. The Prisma schema stores months as 1-indexed integers (matching `Session.month`, `Invoice.month` conventions).

**How to avoid:** In `addMonths`, convert to 0-indexed for arithmetic (`startMonth - 1`), then convert back (`% 12 + 1`). Write unit tests for boundary cases: startMonth=12 (offset=1 ã month=1 of next year), startMonth=10 (offset=5 ã month=3 of next year).

**Warning signs:** Any charge month value outside 1-12, or charges appearing in wrong year.

### Pitfall 4: Relation Not Added to Student Model

**What goes wrong:** `prisma.student.findUnique({ include: { subscriptions: true } })` throws a TypeScript error in Phase 2.

**Why it happens:** Omitting the `subscriptions PlatformSubscription[]` virtual relation field from the `Student` model.

**How to avoid:** The `PlatformSubscription` model references `Student` via `studentId`, but Prisma requires the reverse relation field to be declared on `Student` for `include` to work. Add `subscriptions PlatformSubscription[]` to the `Student` model in the same schema edit.

```typescript
export function addMonths(
  startMonth: number, // 1-indexed
  startYear: number,
  offset: number      // 0-based: 0 = same month, 5 = 5 months later
): { month: number; year: number } {
  const totalMonths = startMonth - 1 + offset;
  return {
    month: (totalMonths % 12) + 1,
    year: startYear + Math.floor(totalMonths / 12),
  };
}
```

### getChargeMonths ã generating all months for a subscription
```typescript
export function getChargeMonths(
  startMonth: number,
  startYear: number,
  durationMonths: number
): Array<{ month: number; year: number }> {
  return Array.from({ length: durationMonths }, (_, i) =>
    addMonths(startMonth, startYear, i)
  );
}
// getChargeMonths(10, 2025, 6)
// ã [{10,2025},{11,2025},{12,2025},{1,2026},{2,2026},{3,2026}]
```

## Open Questions

1. **Does Supabase DIRECT_URL exist in .env.local?**
   - What we know: `schema.prisma` declares `directUrl = env("DIRECT_URL")`, which is correct
   - What's unclear: Whether the developer's `.env.local` actually has `DIRECT_URL` populated
   - Recommendation: Before running `prisma migrate dev`, verify with `printenv DIRECT_URL` or check `.env.local` directly. STATE.md already flags this as a known concern.

2. **Should `billingMethod` be a Prisma enum or a String?**
   - What we know: Existing schema uses `String` for similar categorical fields (e.g., `currency String @default("CHF")`)
   - What's unclear: Using `String` allows typos; a Prisma `enum BillingMethod { invoice direct }` would add type safety at the DB level
   - Recommendation: Use `String` to match existing conventions and keep the migration simple. Application-level TypeScript type (`"invoice" | "direct"`) provides type safety without DB enum overhead.

3. **Should `durationMonths` be constrained to 1 or 6 at the DB level?**
   - What we know: Requirements say "1 or 6 months" but existing schema has no CHECK constraints
   - What's unclear: Whether to add a DB-level CHECK constraint or leave it to application validation
   - Recommendation: Omit DB constraint in Phase 1 (no CHECK constraints exist in current schema). Phase 2 UI will enforce 1 or 6 via form validation.
